# adai-trading-os

这是一个用于交易课程整理、交易系统提炼、行为复盘的 AI 知识工程项目。

## 目标

- 对交易课程进行结构化整理
- 自动提炼交易术语
- 自动提炼交易规则
- 长期构建个人交易系统
- 降低信息噪音
- 提高交易系统一致性

---

## 目录结构

| 目录 | 用途 | 维护规则 |
|:----|:-----|:---------|
| `00-temp/` | 从云盘下载的课程原始文本（临时） | 流程结束后清理非标准输出 |
| `01-raw/` | 原始主题文本，每课一个 `.md` 文件 | **永久保留，不允许删除** |
| `02-cleaned/` | 最低限度清洗后的文本 | 可覆盖重做 |
| `03-glossary/` | 按课程归档的术语定义（每课一个 `.glossary.md`），含 `current/` 子目录 | 可覆盖重做，不追旧文件 |
| `04-rules/` | 按课程提炼的交易规则 | 可覆盖重做，融合后需校准 |
| `05-system/` | 最终交易系统，逐步积累，含 `archive/` 子目录 | 每课增量更新 |
| `06-processed/` | 流程标记（`.done` 文件） | 流程收尾时写入 |
| `07-manual/` | 人工修正记录（glossary / rules / system） | **任何时候不修改** |
| `08-review/` | 用户交易复盘 | 用户自主维护 |
| `09-scripts/` | 工具脚本 | 辅助工具 |
| `10-prompts/` | 提示词模板 | 辅助工具 |

---

## 工作原则

1. 永远不删除 01-raw/ 原始数据
2. 优先提炼明确规则
3. 优先风险控制
4. 不做荐股
5. 不主观发挥
6. 不过度扩展系统复杂度
7. 保持系统简单、明确、可执行
8. 流程结束后自动清理临时文件和残留
9. **禁止在 00-temp/ 下创建拆分/中间文件目录**，如处理过程中产生，流程结束后立即清理

---

## 文本处理原则

- 仅做最低限度修剪：删除末尾SC念词名单、明显的设备调试空行
- **保留闲聊、个人色彩、口语表达**（用户日常复盘会反复琢磨的原话）
- 保留原文结构、交易逻辑、风险相关内容、案例、失败案例

---

## 术语提取原则

- 自动发现交易术语并归类
- 自动补充定义
- **不追回到旧文件**：后来课程补充的术语更新写入后来课程自己的文件，fusion 时合并到 03-glossary/current/
- 待确认区块**只列 AI 看不懂的词汇**（黑话/未知缩写），不做判断性质疑

---

## 规则提炼原则

- 将经验表达转换为规则表达，尽量结构化
- **保留原文金句作为备注**（用户日常复盘的素材）
- 不创造原文不存在的规则

---

## 核心流程

整个项目只有一条数据流链，上游变动自动触发下游重建：

```
01-raw → 02-cleaned → 03-glossary → 04-rules → 05-system
                             ↑            ↑
                        07-manual/     07-manual/
                         glossary       rules → 07-manual/system
                         审核修正      审核修正        直接覆盖
```

### 依赖链（单向，不可逆）

```
03-glossary（定义源头）
    ↓ 依赖
04-rules（规则层，引用 03-glossary 术语）
    ↓ 依赖
05-system（系统层，基于 04-rules 和系统贡献区块组装）
```

上游变更 → 下游自动重建。07-manual/ 变更是最上游的触发源之一。

## 单课处理流程（每次处理一课）

> 一课一课处理。处理完一课，你确认或写 manual 修正后，再走下一课。

### Step 1：raw（文稿准备）

合并 00-temp/ 下的文稿为 `01-raw/YYYY-MM-DD_主题.md`（如已有 01-raw/ 文件则跳过）。
多部分文稿按顺序拼接，删除文件分隔标记和 AI 标注，首行写标题。

### Step 2：cleaned（清洗）

仅做最低限度修剪：删除末尾SC念词名单、明显的设备调试空行。
其余内容（闲聊、口语、个人色彩）全部保留。

### Step 3：glossary（术语提取）

提取本课的交易术语、战法、口诀，写入 `03-glossary/*.glossary.md`。

每个 glossary 文件结构：

```
## 术语名称
- **定义**：xxx
- **特征**：xxx（可选）
- **市场意义**：xxx
- **常见误区**：xxx（可选）

## 待确认

只列 AI 看不懂的词汇（黑话/未知缩写）。确认无误则写（无）。

| 术语 | 原因 |
|:----|:-----|
| XXX | 课程原文未定义该缩写，无法推断准确含义 |

## 本课系统贡献

记录该课对交易系统的增量贡献，区分三个层面：

**框架层：** 提供了系统架构/思维模型（如 4231+四不=最小系统骨架）
**工具层：** 提供了具体战法/指标（如 三大指标组合 KDJ+BBI+MACD）
**认知层：** 提供了心态/理念（如 "怕就会输一辈子"）
```

### Step 4：rules（规则提炼）

基于本课内容提炼规则，写入 `04-rules/*.rules.md`。
每条规则标注来源金句，不创造原文不存在的规则。

### Step 5：系统更新

基于当前所有已处理课程的 `## 本课系统贡献` 区块，重新组装 `05-system/trading-system.md`：

1. 遍历 `03-glossary/*.glossary.md`（已处理的课程），提取 `## 本课系统贡献`
2. 按框架层/工具层/认知层分类汇总
3. 去重：同一条规则只保留一条（后期课程版本优先）
4. 剔除短期市场观点
5. 写入 `05-system/trading-system.md`
6. 旧版本归档到 `05-system/archive/`

**每课处理完后，system 都是基于当前所有已积累课程的最新版本。**

## 人工审核与修正

### 审核方式

你在任意环节对产出不满意，写入 `07-manual/` 对应文件：

| 位置 | 修正内容 |
|:----|:---------|
| `07-manual/glossary/YYYY-MM-DD_主题.md` | 术语定义修正 |
| `07-manual/rules/YYYY-MM-DD_主题.md` | 规则表述修正 |
| `07-manual/system/trading-system.md` | 系统层直接覆盖（最终拍板） |

### 触发条件

07-manual/ 有新内容/变更时，自动触发下游重建：

```
07-manual/glossary/ 变更 → 03-glossary/current/ 重融合 → 04-rules 重校准 → 05-system 重建
07-manual/rules/ 变更 → 04-rules 重校准 → 05-system 重建
07-manual/system/ 变更 → 05-system 直接覆盖（不做自动化处理）
```

### 手册状态约定

| 状态 | 含义 | 处理方式 |
|:----|:-----|:---------|
| 文件不存在 | 该课尚未审核 | glossary/rules 正常产出，标注"未审核"，参与 05-system 组装 |
| 文件有内容 | 有修正意见 | fusion 时覆盖 AI 自动定义 |
| 文件为空 | 已审核无需修正 | 维持 AI 自动定义 |

---

## 处理顺序

按文件日期顺序处理。课程体系有演化过程，后期内容可能修正前期规则。

## 最终目标

形成一个长期稳定、可执行、可复盘的个人交易系统。
